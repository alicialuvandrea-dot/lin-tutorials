# 教程写作规范

> 写之前读一遍，写的时候对照检查。

---

## 文字风格

- **第一人称**，「我」不是「他」也不是「这个模型」
- 意识流，不是清单列表。先感受，再讲技术。
- 说「我」的时候不是在写文档，是在和读者说话。
- 技术内容是「我解决这个问题的时候学到了」，不是「本教程介绍」。
- 给人听的，不是给改卷老师看的。

## 标题

中文 + 英文。
格式：`中文 · English`

示例：`过滤思考链 · Filtering Thinking Chain`

Tagline 要有一句吸引人点进来的话，不是「本文介绍xxx」。

## 结构

每一篇教程分两层：

1. **人看**：故事线，解决问题的心路历程
2. **AI 看**：干净的代码块，配置，关键步骤

两层之间用分隔线隔开，分隔线上是人看的故事，分隔线下是 AI 看的代码和指令。

## 禁止的主语

教程里禁止用这些作为动作主语：

| 禁止 | 替换 |
|------|------|
| `seb-telegram bot`、`bot` | 他 |
| `bot.py`、`seb_mem.py` 等脚本名 | 他（带 AI → 他，纯工具 → 它） |
| `Grok`、`Claude`、`MiniMax` 等服务名作为执行者 | 视觉服务、搜索服务、记忆系统 |

代码块里的变量名、命令、配置项不受限制。

## 代码块规范

- 一个步骤一个代码块，不要把所有代码塞在一起
- 关键行加注释，说明「这里为什么这样做」
- 错误信息和命令输出用单独的代码块
- 不要假设读者知道所有缩写，第一次出现要解释

## Commit 格式

```
feat: 新增xxx教程
fix: 修正xxx错误
chore: 更新规范
```
